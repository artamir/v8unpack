# Редактирование табличной части обычной формы

Документ описывает практический процесс редактирования табличной части обычной формы (например формы документа) в исходниках, распакованных через v8unpack.

---

## Где находится табличная часть

Для формы документа структура обычно такая:

```
Document/<ИмяДокумента>/
└── DocumentForm/
    └── <ИмяФормы>/
        ├── DocumentForm.id.json
        ├── DocumentForm.json
        ├── DocumentForm.obj.bsl
        └── DocumentForm.elem.json
```

Для табличной части формы ключевой файл: `DocumentForm.elem.json`.

---

## Как tabular section хранится в `DocumentForm.elem.json`

По реальной структуре v8unpack файл содержит блоки:

```json
{
  "params": [],
  "props": [ ... ],
  "commands": [ ... ],
  "tree": [ ... ],
  "data": { ... }
}
```

### 1. `tree`

В `tree` табличная часть формы представлена элементом типа `Table`:

```json
{
  "name": "Товары",
  "type": "Table",
  "page": "Страница1"
}
```

### 2. `data`

Для таблицы есть узел по ключу `"Страница/ИмяТаблицы"`, например `"Страница1/Товары"`:

```json
{
  "id": "4",
  "ver": 27,
  "page": "Страница1",
  "raw": [ ... ],
  "prop": "ДокументОбъект"
}
```

- `prop` указывает контекст данных (обычно `ДокументОбъект`).
- Колонки таблицы и визуальные параметры хранятся глубоко внутри `raw`.

### 3. `props`

`props` хранит реквизиты формы. Для табличной части могут быть реквизиты, которые используются таблицей или её обработчиками.

---

## Что реально можно делать вручную

| Задача | Где править | Рекомендация |
|---|---|---|
| Логика заполнения/пересчета строк таблицы | `DocumentForm.obj.bsl` | ✅ Да |
| Обработчики событий таблицы (`ПриИзменении`, команды и т.д.) | `DocumentForm.obj.bsl` | ✅ Да |
| Легкая правка подписей колонок (только текст) | `DocumentForm.elem.json` | ⚠️ Очень осторожно |
| Добавить/удалить колонку, менять тип колонки | `DocumentForm.elem.json` (`raw`) | ❌ Только через Конфигуратор |
| Менять порядок/ширины/привязки колонок | `DocumentForm.elem.json` (`raw`) | ❌ Только через Конфигуратор |

Главный принцип: вручную безопасно править код формы, а макет табличной части править в 1С Конфигураторе.

---

## Рекомендуемый процесс (безопасный)

### Шаг 1. Изменить табличную часть в Конфигураторе

В форме документа (обычная форма):
1. Добавьте/удалите колонки.
2. Измените порядок и свойства колонок.
3. Сохраните конфигурацию.
4. Выгрузите CF.

### Шаг 2. Обновить исходники через v8unpack

```python
import v8unpack

v8unpack.extract('updated.cf', 'src/')
```

В результате `DocumentForm.elem.json` будет согласован с реальной структурой формы.

### Шаг 3. Доработать код формы вручную

Правки логики делайте в `DocumentForm.obj.bsl`, например:
- пересчет итогов по строкам табличной части,
- проверки заполнения колонок,
- обработчики команд формы.

### Шаг 4. Собрать обратно

```python
v8unpack.build('src/', 'patched.cf')
```

### Шаг 5. Проверить в 1С

Откройте форму документа и проверьте:
1. список колонок,
2. порядок и ширины,
3. ввод в строки,
4. работу обработчиков.

---

## Если нужно поменять только код, без изменения макета

Если структура таблицы не меняется:
1. Правьте только `DocumentForm.obj.bsl`.
2. `DocumentForm.elem.json` не трогайте.
3. Собирайте `v8unpack.build(...)`.

Это самый надежный сценарий для CI/CD и автоматических патчей.

---

## Почему не стоит вручную править `raw`

В узле `data["Страница1/Товары"]["raw"]` кодируются:
- свойства таблицы,
- описание колонок,
- привязки,
- координаты,
- часть платформенных служебных структур.

Даже небольшая ошибка в индексах массива приводит к некорректной форме или ошибке загрузки конфигурации.

## Проверенные пути в обычной форме (`CatalogForm`) на реальном примере

Ниже зафиксированы пути, проверенные на форме `Catalog/ТестCF_Справочник_01/CatalogForm/ФормаЭлемента/`.

### 1. Добавление самой таблицы на страницу формы

> Верифицировано на шаге step_016.

При добавлении таблицы формы меняются два файла:
- `CatalogForm.elem.json` — дерево и сам узел таблицы
- `CatalogForm.json` — Max ID и счётчик элементов формы

Точные пути:
- `tree[4].child[0]` — новая запись `{ "name": "ТабличнаяЧасть1", "type": "Table", "page": "Страница1" }`
- `data["Страница1/Панель1/Страница1/ТабличнаяЧасть1"]` — новый узел таблицы
- `data["Страница1/Панель1/Страница1/ТабличнаяЧасть1"]["raw"][2][1]` — связь таблицы с данными ТЧ: `["\"Pattern\"", ["\"#\"", "a3473005-f371-45a4-abe9-1f99c2a1ab48"]]`
- `data["Страница1/Панель1/Страница1/ТабличнаяЧасть1"]["raw"][2][2][1][23][0]` — счётчик колонок таблицы, после создания пустой таблицы равен `"0"`
- `form[0][0][1][1][1]` — Max ID формы (`"13"` → `"14"`)
- `form[0][0][1][10]` — счётчик элементов формы (`"6"` → `"7"`)

### 2. Где именно лежат колонки таблицы формы

> Верифицировано на шагах step_017-step_020.

Колонки таблицы хранятся по пути:

`data["Страница1/Панель1/Страница1/ТабличнаяЧасть1"]["raw"][2][2][1][23]`

Структура блока:
- `[23][0]` — количество колонок
- `[23][1]` — первая колонка
- `[23][2]` — вторая колонка
- `[23][3]` — третья колонка
- `[23][4]` — четвертая колонка

На проверенном примере:
- step_017: `[23][0]` `"0"` → `"1"`, добавлена системная колонка `НомерСтроки` в `[23][1]`
- step_018: `[23][0]` `"1"` → `"2"`, добавлена колонка `Реквизит1Строка10Переменная` в `[23][2]`
- step_019: `[23][0]` `"2"` → `"3"`, добавлена колонка `Реквизит2Число10_2_неотрицательное` в `[23][3]`
- step_020: `[23][0]` `"3"` → `"4"`, добавлена колонка `Реквизит3Справочник2` в `[23][4]`

У каждой колонки внутри дескриптора используются стабильные индексы:
- `[n][1][1][1][30]` — имя/привязка колонки
- `[n][1][1][1][35]` — `Pattern` типа колонки
- `[n][1][1][1][38]` — UUID хранилища обработчиков колонки
- `[n][1][1][1][39][0][0]` — base64 blob со связями обработчиков

Проверенные значения `Pattern`:
- `НомерСтроки`: `["\"Pattern\""]`
- `Реквизит1Строка10Переменная`: `["\"Pattern\"", ["\"S\"", "10", "1"]]`
- `Реквизит2Число10_2_неотрицательное`: `["\"Pattern\"", ["\"N\"", "10", "2", "1"]]`
- `Реквизит3Справочник2`: `["\"Pattern\"", ["\"#\"", "1034dc46-97b9-4466-a9b8-4e1569c6513e"]]`

После добавления каждой колонки инкрементируется `form[0][0][1][10]` в `CatalogForm.json`:
- step_017: `"7"` → `"8"`
- step_018: `"8"` → `"9"`
- step_019: `"9"` → `"10"`
- step_020: `"10"` → `"11"`

### 3. Где хранится обработчик события колонки

> Верифицировано на шаге step_021.

Для обработчика `ПриИзменении` у колонки меняются два места:

1. В `CatalogForm.obj.bsl` добавляется процедура формы, например:

```bsl
Процедура ТабличнаяЧасть1Реквизит2Число10_2_неотрицательноеПриИзменении(Элемент)
  // Вставить содержимое обработчика.
КонецПроцедуры
```

2. В `CatalogForm.elem.json` меняется base64 blob у конкретной колонки:

`data["Страница1/Панель1/Страница1/ТабличнаяЧасть1"]["raw"][2][2][1][23][3][1][1][1][39][0][0]`

Для шага step_021 это дескриптор колонки `Реквизит2Число10_2_неотрицательное`.

Важно:
- UUID хранилища события в `[38]` не обязан меняться; в проверенном примере остался `381ed624-9217-4e63-85db-c4c3cb87daae`
- визуальная структура таблицы не меняется, но `CatalogForm.json` всё равно инкрементирует внутренний счётчик `form[0][0][1][10]` (`"11"` → `"12"`)

## Быстрый чек-лист

1. Меняется структура табличной части (колонки, порядок, видимость)?
   - Делать только в Конфигураторе.
2. Меняется только поведение (проверки, вычисления, команды)?
   - Править `DocumentForm.obj.bsl`.
3. После изменений всегда:
   - `extract` (если меняли в конфигураторе),
   - `build`,
   - проверка в 1С.
